# 文档管理逻辑审查报告

## 概述

本文档对 uniCode 项目的文档管理系统进行系统性代码审查，覆盖审核操作、文档索引、数据库层、前端页面、CLI 工具五个模块。目标读者为项目维护者，用于指导后续 bug 修复和代码优化。

审查范围：
- `docs/lib/actions.ts` — 审核操作（approve/reject/unpublish/resubmit/batchApprove）
- `docs/lib/index.ts` — 文档读取与聚合
- `docs/lib/db/docs.ts` — 数据库 CRUD
- `docs/lib/db/schema.ts` — SQLite schema 与连接管理
- `docs/lib/types.ts` — 类型定义
- `docs/app/*.tsx` — 前端页面与组件
- `docs/tools/*.ts` — CLI 工具（seed, publish-docs, scan-docs, check-code-refs）

## 正文

### 一、严重 Bug

#### BUG-1: `unpublishDoc` 后 DB 文档仍显示在正式文档列表中

**位置**: `docs/lib/actions.ts:451-481` + `docs/app/layout.tsx:77`

**问题**: `unpublishDoc` 将 DB 文档的 status 改为 'draft'，但文档仍在 SQLite 数据库中（`relativePath` 保持空字符串）。`layout.tsx` 中正式文档的过滤条件是 `!d.relativePath`，因此退回草稿状态的文档仍出现在 `/docs` 正式文档列表中，而不会出现在 `/docs/drafts` 草稿区。

**根因**: 正式/草稿的判别标准是 `relativePath` 是否为空，而非 `status` 字段。`unpublishDoc` 没有将文档从 DB 删除并写回文件系统。

**修复方向**: `unpublishDoc` 应像 `rejectDoc` 一样，将 DB 文档删除并写回 `docs/drafts/` 文件系统，同时添加反向重定向。

---

#### BUG-2: `ReviewRecord.action` 类型与实际写入值不一致

**位置**: `docs/lib/types.ts:6` vs `docs/lib/actions.ts:205`

**问题**:
- types.ts 定义 `action: 'approved' | 'rejected' | 'published' | 'unpublished'`
- `approveDoc` 第 205 行实际写入 `action: 'published'`
- `ACTION_LABELS` 虽然有 `approved: '通过审核'` 映射，但 `'approved'` 永远不会被任何代码写入

types.ts 中的 union 类型包含了从未使用的 `'approved'` 字面量，类型与实际行为脱节。

**修复方向**: 统一术语 — 要么从 types.ts 移除 `'approved'`，要么将 `approveDoc` 中的 `'published'` 改为 `'approved'`。

---

#### BUG-3: `batchApproveDocs` 重复调用 `publishDocs()`（N+1 问题）

**位置**: `docs/lib/actions.ts:343-359`

**问题**: `batchApproveDocs` 循环调用 `approveDoc(slug, '')`，每个 `approveDoc` 内部已经调用了一次 `publishDocs()`。循环结束后又额外调用一次。对于 N 篇文档，共调用 N+1 次 `publishDocs()`。每次 `publishDocs()` 全量扫描 DB 中所有 published 文档并写入文件系统。

**修复方向**: 从 `approveDoc` 中移除 `publishDocs()` 调用，统一由调用方在操作完成后调用一次。

---

#### BUG-4: `scan-docs.ts` 不扫描数据库中的正式文档

**位置**: `docs/tools/scan-docs.ts`

**问题**: `scan-docs.ts` 只扫描 `docs/` 目录下的 `.md` 文件，但正式文档已迁移到 SQLite 数据库。后果：
1. 数据库文档的 frontmatter 合规性不被检查
2. 数据库文档中的死链不被检测
3. `collectAllSlugs` 缺少数据库 slug，导致文件→DB、DB→文件、DB→DB 的交叉引用无法验证

**修复方向**: 加入 DB 文档扫描，合并 DB + 文件系统的 slug 集合进行全量引用检查。

---

### 二、中等缺陷

#### DEFECT-1: `approveDoc` 晋升时全局文本替换可能误伤

**位置**: `docs/lib/actions.ts:264-277`

草稿晋升为正式文档时，扫描项目所有文本文件，用 `String.split(old).join(replacement)` 做全局替换。如果 oldSlug 是 `index` 这样的常见词，可能错误替换不相关的出现。

**修复方向**: 使用正则做单词边界匹配，或限定替换范围为 markdown 链接语法 `](...)` 内部。

---

#### DEFECT-2: 驳回失败时输入框仍然关闭

**位置**: `docs/app/[...slug]/review-button.view.tsx:101`

```typescript
onClick={() => handle(() => rejectDoc(slug, rejectReason)).then(() => setShowReject(false))}
```

`handle()` 内部 catch 了所有错误且不 rethrow，所以 `.then()` 永远执行。驳回失败时输入框也关闭了，用户看不到失败原因就被打断。

**修复方向**: 只在操作成功时关闭输入框。

---

#### DEFECT-3: 文档详情页未显示 `target_path`

**位置**: `docs/app/[...slug]/page.tsx:67-100`

对于草稿文档，`target_path` 是核心元数据（标明发布后的目标路径），但详情页完全不显示。审核人在决定通过时看不到目标位置。

**修复方向**: 在文档信息栏中增加 `target_path` 显示（draft 状态 + target_path 非空时展示）。

---

#### DEFECT-4: 正式文档列表 `last_reviewed` 空值时语义不清

**位置**: `docs/app/page.tsx:102`

```typescript
{doc.isStale ? <span>过期</span> : doc.frontmatter.last_reviewed || '-'}
```

对于从未复审过的文档，显示 `-`，没有 "未复审" 语义提示。

---

#### DEFECT-5: `writeFrontMatter` 可能改变手动编写的 YAML 格式

**位置**: `docs/lib/actions.ts:109-120`

`writeFrontMatter` 使用 `matter.stringify(content, ordered)` 重新序列化 YAML frontmatter。`gray-matter` 可能改变用户手写注释或多行字符串的格式。此函数在 `approveDoc` 晋升后更新引用时被调用。

---

### 三、代码质量问题

#### QUALITY-1: `STATUS_CONFIG` 在 4 个文件中完全重复

| 文件 | 行号 |
|------|------|
| `docs/app/page.tsx` | 4-9 |
| `docs/app/[...slug]/page.tsx` | 14-19 |
| `docs/app/drafts/page.tsx` | 5-10 |
| `docs/app/drafts/batch-approve.view.tsx` | 7-12 |

每次修改状态样式需要同步 4 处。

---

#### QUALITY-2: `DIR_LABELS` 在 2 个文件中定义且结构不同

- `docs/app/layout.tsx:17-29` — 单层 key（如 `'development'`, `'testing'`）
- `docs/app/page.tsx:11-20` — 多层 key（如 `'guides/development'`, `'guides/testing'`）

两份标签映射结构不一致，维护时容易遗漏。

---

#### QUALITY-3: 草稿区树结构完全扁平无分组

**位置**: `docs/app/layout.tsx:31-72` + `docs/tools/seed.ts`

种子脚本使用平级文件名（如 `architecture-adr-adr-0001-nextjs-app-router.md`）。`buildTree` 对该文件名的 `parts` 只有一个元素，14 篇种子文档在草稿区侧边栏中毫无结构。

**修复方向**: 使用 `target_path` 字段构建草稿区的层级树，预览发布后的目录结构。

---

#### QUALITY-4: `getAllDocs` 合并逻辑中 `dbSlugs` 检测无实际意义

**位置**: `docs/lib/index.ts:90-99`

文件系统文档的 slug 格式为 `drafts/xxx`，DB 文档 slug 格式为 `path/to/doc`。两者命名空间不重叠，`dbSlugs.has(fd.slug)` 永远为 false，Set 检测形同虚设。

---

### 四、潜在风险

#### RISK-1: draft 文档若无 `target_path`，approve 行为模糊

**位置**: `docs/lib/actions.ts:208`

```typescript
if (entry.relativePath && entry.frontmatter.target_path) {
```

草稿文档通过审批时，若忘记设置 `target_path`，条件不成立，会走到下面的 "DB 文档—直接改状态" 分支。但这是文件系统草稿（DB 中不存在），`saveReviewData` 会写回文件，结果是 status 变成 'published' 但文档仍留在 drafts 目录，既不在 DB 也不在生产快照中。

---

#### RISK-2: `publish-docs.ts` 的 CLI 检测在 `npx tsx` 下可能失效

**位置**: `docs/tools/publish-docs.ts:113-114`

`process.argv[1]` 在某些环境下为 `undefined`。`npx tsx` 可能不设置 `argv[1]` 为脚本路径，导致 `runningDirectly` 为 false，CLI 静默无输出。

---

#### RISK-3: WAL 文件持续增长

`closeDb()` 导出后只在 `seed.ts` 的 `clearDb()` 中调用。其他 CLI 工具都不执行 WAL checkpoint，Write-Ahead Log 文件会持续累积。

---

#### RISK-4: `_redirects.json` 重定向链 > 1 无法检测

当前位置处的防循环清理只删除**直接**相关的条目。对于长度 > 1 的链，无法检测。但在当前 draft↔published 两种 slug 参与重定向的模式下，链长不太可能超过 2，风险较低。
