# Next.js全栈开发规范

## 架构与工程化规范
1. 核心业务代码必须统一收敛于独立的源代码目录中，实现业务逻辑与配置文件的物理隔离，严禁将业务代码散落于项目根目录。
2. 面对包含网页端、移动端和桌面端的多端需求，必须采用单一代码仓库（Monorepo）架构。将通用的业务逻辑、类型定义和工具函数提取为共享包，实现底层逻辑的单点维护和跨端复用。
3. 全栈框架必须作为网页端核心并提供内部服务端动作，同时暴露标准的外部通信接口供移动端和桌面端调用，实现"一次后端开发，多端无缝接入"。
4. 项目启动前必须确立唯一的真实来源。数据库的表结构、关联关系和索引定义必须拥有唯一的核心模型配置文件，所有的服务端与客户端类型推断均需从此文件自动生成，严禁手动维护重复的类型定义。

## 目录与文件组织规范
1. 路由目录不仅代表页面访问路径，更应作为业务领域边界。每个路由文件夹应视为一个独立的业务模块，严禁不同业务模块之间的底层逻辑强耦合。
2. 业务逻辑处理文件（如数据获取、工具函数、自定义钩子）必须放置在路由目录同级或专属的公共依赖目录中，严防被框架误识别为公开的网页路由。
3. 全局单例服务、核心工具库、第三方集成（如数据库连接、统一鉴权、审计日志）必须集中存放在专用的公共依赖目录中，以便全局统一调用。
4. 所有的静态资源（如图标、占位图、基础样式文件）必须归拢于专用的公共资源目录，并通过绝对路径引用。

## 命名约定与代码风格规范
1. 所有自定义目录与文件必须严格使用中划线命名法（小写字母加连字符），彻底杜绝跨操作系统协作时引发的大小写敏感致命错误。
2. 必须严格遵循全栈框架的路由保留字规范。如页面视图、布局、加载占位、错误边界以及接口路由等特殊文件，必须使用框架规定的标准保留字进行命名。
3. 组件文件必须以清晰的后缀结尾，区分其是普通视图组件、布局组件还是错误边界组件。
4. 服务端接口处理文件和数据库模型操作文件，必须在命名上明确其职责领域，保持高可读性和可追溯性。
5. 变量名与函数名必须具备强烈的自解释性。布尔值变量必须以表达状态的动词开头，事件处理函数必须以标准前缀开头。

## 前端视图与组件规范
1. 组件库必须严格区分"原子化展示组件"与"复合型业务组件"。底层展示组件绝不允许包含任何服务端交互、数据获取或复杂的全局状态读写，必须保持纯粹的输入输出特性。
2. 针对高度动态的界面渲染（如基于热度生成的复杂三维图形或动画），必须在运行时使用纯算法实时计算并生成纹理与轨迹，严禁依赖静态预设的高内存图片资源。
3. 所有的用户交互必须包含明确的加载状态反馈、成功提示与失败回退机制。严禁在数据提交期间让用户处于无响应的等待状态。
4. 页面布局必须采用现代化的弹性盒子或网格布局方案，确保在任意屏幕尺寸和设备下，内容区块均能自适应排列且不发生遮挡与溢出。

## 状态管理与数据流规范
1. 前端数据流向必须保持单向可控。对于服务端获取的数据，优先依赖框架自带的缓存与重新验证机制，避免在客户端引入沉重的全局状态管理库。
2. 跨组件传递的深层数据，必须通过上下文（Context）机制进行作用域隔离，严禁无意义的逐层属性透传。
3. 凡是由前端页面交互触发的服务端数据变更，必须统一使用框架原生的服务端动作机制，彻底消除传统的网络请求样板代码，实现数据变更后的自动视图失效与刷新。

## 路由与通信接口规范
1. 前后端通信接口必须实行双轨制隔离设计：面向内部网页组件的交互采用强类型的服务端动作，面向外部移动端、桌面端或第三方的通信采用标准状态码的接口。
2. 服务端动作必须实现强类型契约。前端调用时必须能够直接从引入的函数中推断出入参结构和返回值的确切类型，做到代码即接口文档。
3. 所有的外部通信接口在处理跨域或嵌套访问时，必须动态处理并剥离不必要的安全限制响应头，并在必要时强行注入基准路径以保证静态资源与接口的正常加载。
4. 外部通信接口必须严格遵循标准的请求方法语义，数据的获取、创建、全量更新、局部更新与删除必须对应正确的网络请求动词。

## 数据库与数据模型规范
1. 数据库模型设计中必须广泛且精准地使用级联删除机制。确保当主实体被删除时，所有相关的弱实体（如绑定的文件记录、评论、点赞）会自动被数据库引擎清理，彻底杜绝孤儿数据与脏数据。
2. 涉及高频查询的字段、外键关联字段以及用于排序和过滤的字段，必须在模型定义层面建立合适的索引，保障查询性能。
3. 所有的数据库查询操作必须精确指定需要返回的字段，严禁在不需要全量数据时使用全字段查询，以最小化内存占用和网络传输开销。
4. 数据库环境（本地开发与线上生产）的切换必须通过环境变量动态挂载连接字符串实现，绝对禁止在代码中硬编码任何数据库连接信息。

## 身份认证与权限管理规范
1. 权限系统必须基于动态会话机制构建。用户登录后，系统必须在每一次会话刷新时，从数据库实时读取最新的角色与状态信息，真正做到权限变更零延迟生效。
2. 严禁使用长期有效的静态权限令牌。所有的服务端接口在执行核心业务逻辑前，必须将权限校验作为第一道防线，显式断言当前调用者的身份与角色。
3. 必须在全局层面实施客户端僵尸会话检测。确保当用户在数据库中被物理删除或管理员禁用时，其前端残留的登录状态和本地凭证能够被强制且静默地清退。
4. 对于复杂的业务实体，必须支持微观的细粒度权限控制。不仅要校验用户的全局角色，还必须校验其是否为当前操作实体的合法所有者或被授权人。

## 资源处理与文件上传规范
1. 接口在处理大文件上传或复杂数据流转换时，必须采用流式读取和分块处理机制，严禁将完整文件一次性加载至内存中，严格控制内存边界，彻底杜绝内存溢出风险。
2. 系统必须具备动态的文件伺服与安全控制网关。绕过静态文件服务器的死板限制，动态读取本地或云端的上传文件。
3. 所有的文件访问路由必须内置防目录穿越保护，强制拦截包含特定上级目录符号的非法请求。
4. 所有的文件上传与读取必须实施严格的目录与类型白名单校验。仅允许符合业务逻辑的指定类型目录被访问，并根据文件后缀名动态赋予正确的响应类型头。
5. 非核心业务的输入输出操作（如审计日志写入、远程备份上传）必须采用异步非阻塞设计，并通过异常捕获包裹，确保决不会因为附属功能失败而中断主业务流程。

## 安全防护与防御性编程规范
1. 凡是能够通过自动识别、自动获取或自动适配来完成的内容（如通过请求头动态提取真实IP地址或设备信息），都必须采用动态处理方式。
2. 严禁硬编码。严禁预填固定值。严禁把参数、路径、环境信息或资源位置写死为静态配置。程序中涉及的所有前置条件，必须在运行时通过检测、提取和加载机制完成处理。
3. 所有来自客户端的数据，无论是服务端动作入参、外部接口请求体，还是统一资源定位符中的查询参数，必须经过强类型校验工具进行严格过滤与清洗，禁止将未经校验的数据直接送入数据库引擎。
4. 代码在运行结束之前，必须清理并回收全部已申请的系统资源，包括但不限于内存、文件句柄、网络连接、锁以及其他临时对象。禁止出现资源泄漏、溢出、无效持有或未注销引用的情况。

## 性能优化与运行时规范

### 异步数据获取与请求瀑布消除（CRITICAL——性能的头号杀手）

1. 所有相互独立、不存在数据依赖的异步操作（如并行获取用户信息、文章列表与评论数据），必须通过 Promise.all() 或 Promise.allSettled() 并发执行，严禁以串行 await 链式调用，将多次网络往返压缩为单次往返。
2. 对于包含部分依赖关系的异步调用，必须将无依赖的操作提前并发启动、有依赖的操作尾随其后，避免简单粗暴的全串行化。凡是可以提前发出的请求，必须在函数入口处立即创建 Promise，将 await 延迟到实际使用该值的代码行。
3. 在执行异步远程判断（如特性开关、权限标记、远程配置查询）之前，必须优先评估本地的同步条件（如组件属性、请求元数据、已加载状态）。若同步条件已经判定该分支不可达，则必须跳过远程异步调用，避免在冷路径上浪费网络请求。
4. 凡是 await 操作所产出的数据仅在特定条件分支内使用时，必须将 await 移至该分支内部，严禁在分支判断之前提前等待。当提前返回路径被频繁触发时，此优化尤为关键。
5. 在服务端接口路由中，必须在函数入口处立即启动所有可并行的 Promise，将 await 延迟至响应返回前的最后一刻，不得在中间步骤上浪费并发的窗口期。

### 流式渲染与 Suspense 边界优化（CRITICAL——直接影响首次内容绘制）

1. 禁止在页面级异步组件中等待全部数据就绪后才返回 JSX。必须使用 Suspense 边界将数据依赖组件隔离，使外壳 UI（导航栏、侧边栏、页脚等静态区域）在数据加载期间立即可见，仅对数据依赖区域显示骨架屏占位。
2. 需要相同数据源的多个组件，必须在父级提前创建数据获取 Promise 并以 props 传递，各子组件通过 React.use() 解包同一 Promise，确保一次请求供多个组件共享，彻底消除重复请求。
3. 对于不影响页面布局决策的非关键数据区域，必须优先使用 Suspense 流式渲染；对于会影响布局稳定性的首屏关键内容或 SEO 敏感区域，谨慎评估 Suspense 的布局偏移代价后再决定是否采用。

### 打包体积与模块加载优化（CRITICAL——直接影响可交互时间与最大内容绘制）

1. 严禁从包含海量重导出的桶状入口文件（如 icon 库、组件库的顶层 index）中直接导入模块。必须在 next.config.ts 中启用 experimental.optimizePackageImports 配置项，将 lucide-react、recharts、date-fns 等重型库纳入自动优化列表，由框架在构建时将桶状导入转换为精确的子路径导入。
2. 针对非首屏关键路径的大型组件（如代码编辑器、富文本编辑器、图表组件、视频播放器），必须使用 next/dynamic 实现按需动态加载，将重型模块从初始脚本包中彻底分离。对于纯客户端组件，必须显式设置 ssr: false 以避免服务端无意义的模块解析。
3. 第三方分析统计、用户行为追踪、错误上报日志等非用户交互关键路径的脚本，必须使用 next/dynamic 配合 ssr: false 推迟至客户端水合完成后再加载，严禁它们阻塞主业务脚本的下载、解析与执行。
4. 仅在特定功能被激活后才需要的大体积模块（如动画帧序列、3D 模型数据、复杂滤镜），必须在用户触发启用该功能时（useEffect 中检测特定条件）才执行动态 import()，并始终通过 typeof window !== 'undefined' 守卫阻断服务端打包。
5. 对于用户可能高频触发的重型功能入口（如"打开编辑器"按钮），必须在按钮获得焦点或鼠标悬停时提前预加载对应模块（利用 import() 的副作用触发浏览器预取缓存），以减少用户点击后的可感知延迟。

### 服务端渲染性能优化（HIGH——消除服务端瓶颈）

1. 所有在同一请求生命周期内被多个组件或函数重复调用的数据库查询、权限校验、文件系统操作，必须使用 React.cache() 进行请求级去重包装。被 cache() 包裹的函数在单次请求内多次以相同参数调用时，仅执行一次，其余调用直接返回缓存结果。注意：React.cache() 使用浅比较判定缓存命中，因此被缓存函数的参数必须为原始值（字符串、数字、布尔值），严禁传递内联对象或数组作为参数。
2. 对于需要跨越多次独立请求共享的数据（如用户短期内依次访问多个页面均涉及同一用户信息），必须在 React.cache() 之上叠加 LRU 缓存层（如 lru-cache 库），设置合理的最大条目数与生存时间。在 Vercel Fluid Compute 等长运行实例环境下，该缓存天然跨请求共享；在传统 Serverless 短实例环境下，需评估是否引入 Redis 等外部缓存。
3. 在 React 服务端组件向客户端组件传递数据时，必须精确控制序列化边界。仅传递客户端组件实际需要的字段，禁止将整个数据库实体对象透传至客户端。客户端组件使用一个字段，服务端就只传递一个字段，这直接减少内嵌于 HTML 中的 RSC 载荷体积。
4. 当需要将同一个数据集的多种视图（如原始列表与排序后列表）传递给客户端时，严禁在服务端同时传递两份独立数组（这会触发两次完全序列化）。必须在服务端只传递一份数据，在客户端通过 useMemo 完成排序或过滤等衍生计算。
5. 所有不直接影响 HTTP 响应体的附属操作（如审计日志记录、用户行为分析、非关键通知推送、缓存标记失效），必须使用 Next.js 的 after() 函数包裹，使其在响应发送至客户端之后异步执行，不阻塞用户的等待时间。

### 组件渲染与重渲染优化（MEDIUM——减少无意义的重复计算）

1. 凡是包含高开销计算（如复杂数据转换、大列表排序过滤、计算密集型派生值）并作为父组件子节点的模块，必须提取为独立组件并以 React.memo() 包裹，使父组件在与此子组件无关的状态变更时跳过该子树的递归渲染。注意：如果项目已启用 React Compiler，手动 memo 不再必要，编译器将自动完成优化。
2. 严禁在组件函数体内部定义另一个组件。在父组件渲染函数内定义的子组件，每次渲染时其组件类型引用均不同，React 将视其为全新的组件类型并执行完整的卸载/挂载流程，导致内部状态全部丢失、副作用重复执行、DOM 节点重创建。所有子组件必须在模块顶层独立定义，需要的数据通过 props 传递。
3. 凡是高频触发但不需要立即反馈至界面的状态更新（如滚动位置追踪、窗口尺寸变化响应、实时搜索输入），必须使用 startTransition 将其标记为低优先级过渡更新，确保用户的直接交互（如输入、点击）不会被派生渲染阻塞。
4. 当用户输入触发对大型数据集的前端搜索或过滤并导致渲染卡顿时，必须使用 useDeferredValue 将原始输入值标记为可延迟值。React 将优先响应用户输入更新，待主线程空闲时再更新过滤结果。同时必须将耗时的过滤逻辑包裹在 useMemo 中，并在过滤结果与原始值不一致时通过透明度或加载指示器向用户传达"结果更新中"的状态。
5. 使用 useState 时，若初始值需要从 localStorage、sessionStorage 读取，或需要执行开销较大的数据结构构建（如建立搜索索引、Map 映射表），必须向 useState 传入初始化函数而非直接传入值。直接传值将导致初始化逻辑在每次渲染时均被执行（尽管结果仅首次被采纳），传入函数则确保其仅在组件挂载时执行一次。
6. 当一个 useMemo 或 useEffect 内部包含多项互相独立的子任务且各自依赖不同的变量时，必须将其拆分为多个独立的 Hook。不得将过滤逻辑与排序逻辑合并在一个 useMemo 中（排序条件变化会导致过滤也重算），不得将页面埋点与标题更新合并在一个 useEffect 中（标题变化会导致埋点重复上报）。
7. 凡是基于当前 state 推算新 state 的 setState 调用，必须使用函数式更新形式（setState(prev => newState)），而非直接引用 state 变量。函数式更新可消除对 state 的依赖，使 useCallback 保持稳定引用不再随 state 变化而重建，同时从根本上杜绝因闭包捕获过期 state 值导致的陈旧闭包缺陷。
8. 所有由用户具体交互动作（点击提交、按下按钮、拖拽释放）所触发的副作用，必须在事件处理函数中直接执行，严禁通过 useEffect 监听一个为跟踪交互而设的标志状态变量来间接触发。将交互建模为"状态变更 + effect 响应"会导致 effect 在不相关依赖变更时重复执行，并引入难以追踪的副作用时序问题。
9. 若某动态状态（如 useSearchParams、localStorage 值）仅在事件回调中被读取而在 JSX 渲染中不被使用，则不得通过 Hook 订阅该状态。必须在回调函数内部按需读取（如在点击事件中直接从 window.location.search 解析参数、从 localStorage 中即时取值），避免因无关状态变更导致组件的不必要重渲染。
10. 凡是内容不随组件渲染而变化的静态 JSX 片段（如静态 SVG 图标、固定骨架屏结构、不变的页脚），必须提升至模块顶层定义为常量，不得在每次组件渲染时重新创建新的 React 元素对象。
11. 条件渲染中，当判断条件的值可能为数字 0、NaN 或空字符串等 falsy 值时，必须使用显式三元表达式（condition ? <Component /> : null）取代短路求值运算符（condition && <Component />），防止 0 或 NaN 被直接渲染到 DOM 中。
12. 对于频繁切换可见性且内部包含昂贵渲染成本的组件（如下拉菜单、选项卡面板、模态框），必须使用 React 的 Activity 组件配合 mode 属性（visible / hidden）实现显隐控制，而非通过条件渲染直接卸载/挂载。Activity 在隐藏时保留 DOM 节点与组件状态，恢复可见时无需重建。

### 渲染性能与浏览器绘制优化（MEDIUM——降低浏览器渲染负担）

1. 对于渲染超过数百条项目的长列表或无限滚动容器，必须对其列表项应用 CSS 的 content-visibility: auto 属性并配合 contain-intrinsic-size 预估高度。浏览器将跳过对屏幕外项目的布局与绘制计算，使初始渲染速度获得数量级提升。
2. 在 SSR 或 RSC 渲染中，凡是依赖客户端存储（localStorage、cookie）决定初始渲染结果的场景（如主题暗亮模式、用户偏好语言），严禁使用 useEffect 在客户端挂载后再修正，这会造成肉眼可见的闪烁。必须在服务端返回的 HTML 中嵌入同步执行的内联脚本（通过 dangerouslySetInnerHTML 注入 IIFE），在浏览器解析 HTML 后、React 水合前直接将正确值写入 DOM，确保水合一致性且无视觉跳变。
3. 对 SVG 图标的动画交互，必须将 CSS 动画或 transform 应用于包裹 SVG 的 div 容器元素，而非直接作用于 SVG 根元素上，以获得更稳定的跨浏览器渲染性能。
4. 超大或高精度 SVG 路径坐标数据应在构建阶段预先进行精度压缩（保留合理的有效数字位数），减少 DOM 节点体积和解析开销。
5. 使用 Suspense 管理加载状态时，优先使用 useTransition Hook 获取 isPending 标志来渲染加载指示器，而非自定义 isLoading 状态配合 useEffect 管理，以确保加载状态与 React 并发渲染机制保持一致的时序。
6. 对于浏览器在初始 HTML 解析阶段就需要预连接或预加载的关键第三方域名（如 CDN、图片服务、API 源站），必须通过 React DOM 的资源提示组件或 Next.js 的 Metadata API 声明式地在文档头部注入 preconnect、dns-prefetch 或 preload 标签。
7. 所有非首屏关键路径的客户端脚本标签，必须显式添加 defer 或 async 属性，禁止同步阻塞 HTML 解析。内联脚本默认即为同步阻塞，除非功能确实需要在解析到该标签时立即执行，否则应避免使用。

### 客户端数据获取与网络请求规范（MEDIUM-HIGH——减少冗余网络开销）

1. 客户端组件中涉及向相同 API 端点发起请求的多个实例，必须统一使用 SWR（或等效的请求去重与缓存库）进行数据获取。SWR 自动将并发请求合并为单次网络调用，并提供缓存共享、过期重新验证、焦点刷新等开箱即用的能力。严禁在每个组件实例中通过 useEffect + fetch 自行管理数据，这导致相同端点被重复请求且无缓存复用。
2. 对于加载后不再变化的数据（如全局配置、静态字典），必须设置不可变模式（如 SWR 的 revalidateIfStale: false 或 useImmutableSWR），禁止进行不必要的后台重新验证。
3. 所有注册在 document 或 window 上的全局滚动、触摸、滚轮事件监听器，若其回调函数不调用 preventDefault()（即仅用于埋点统计、日志记录、状态追踪等非阻断性用途），必须附加 { passive: true } 选项。浏览器在遇到非被动监听器时须等待其执行完毕方可继续滚动绘制，passive 选项使浏览器可立即响应滚动，从根本上消除因事件监听导致的页面滑动延迟。
4. 存储在 localStorage 或 sessionStorage 中的结构化数据必须附带版本号字段，读取时先校验版本号再解析内容，结构变更时自动降级为默认值，防止因数据格式升级导致的客户端解析崩溃。同时应禁止存储超过数 KB 的大体积数据到 Storage 中，大文件应走 IndexedDB 或服务端持久化。

### JavaScript 运行时微优化（LOW-MEDIUM——积少成多的热路径收益）

1. 对需要反复按键查找的线性遍历场景（如在事件列表中按事件 ID 查找），必须预先构建 Map 或 Set 数据结构，将 O(n) 的重复遍历降至 O(1) 的哈希查找。在循环体内进行数组的 includes() 或 indexOf() 调用是此问题的典型反模式。
2. 对同一数组需要同时执行过滤和映射两步操作时，必须使用 flatMap 在一次遍历中完成，而非先 filter 再 map（两次遍历）。同理，需要先过滤再排序时，应优先合并链路减少遍历趟数。
3. 纯计算型函数（输入到输出的确定性映射，无副作用）在热路径中被反复以相同参数调用时，必须在模块顶层维护一个 Map 作为结果缓存，以参数为键、结果为值，命中缓存时直接返回，避免重复计算。
4. 函数内部存在多个卫语句（Guard Clause）进行异常或边界条件判断时，必须将最可能成立的条件放在函数体最前面，使得失败路径尽早退出、成功路径尽量减少嵌套层级。
5. 正则表达式对象若在循环体或高频回调内使用，必须将其定义提升至循环外部、模块顶层或 Hook 外部，避免每次迭代或每次渲染时重复编译正则表达式。
6. 任何读取 localStorage 或 sessionStorage 的操作，若在同一函数调用栈或同步渲染栈中被多次访问同一键名，必须将其结果缓存到局部变量中，严禁重复执行同步的磁盘 I/O 读操作。
7. 对 DOM 元素需要连续设置多个 CSS 样式属性时，必须批量操作：优先切换 class 类名（一次性应用一组样式），其次使用 cssText 批量赋值，最后才考虑逐个修改 style 属性，以最小化样式重计算的触发次数。
8. 在对两个数组进行全量比较（如判断数据是否变化）之前，必须先比较两者的 length 属性。长度不同则立即返回不相等，避免进入昂贵的逐项深度比较。
9. 对于无需响应用户交互的后台任务（如预渲染不可见画布帧、预计算缓存数据、批量预处理），必须使用 requestIdleCallback 将其延迟至浏览器空闲时段执行，确保不挤占主线程上的用户交互、动画渲染等关键任务。
10. 在一组数据中查找最大值或最小值时，必须使用单次 for 循环遍历并维护当前极值，严禁先调用 sort() 排序再取首尾元素——排序的 O(n log n) 开销远大于单次遍历的 O(n)。

### 高级并发与性能模式（LOW——特定场景的精准优化）

1. 对于仅在组件初始化时需要执行一次的高开销操作（如建立 WebSocket 连接、初始化第三方 SDK、注册全局拦截器），必须保证其在应用整个生命周期内仅执行一次。通过模块顶层的布尔锁或惰性单例模式实现，不依赖 useEffect 或多次挂载的组件生命周期。
2. 需要在对同一客户端组件的多次快速状态更新均被完整处理（而非被 React 的自动批处理合并）的场景中，可使用 flushSync 强制执行同步渲染刷新。此 API 应极度审慎使用，因其会破坏 React 的并发调度优化并导致性能退化，仅当 DOM 操作存在强制性时序依赖（如滚动到新添加元素的确切位置后再触发焦点切换）时才可调用。
3. 必须预装并验证项目是否已启用 React Compiler（React 19 的实验性特性）。若已启用，则本规范中所有手动 memo、useMemo、useCallback、memo() 的优化规则可自动放宽——编译器会在构建阶段自动识别并插入等价优化。但函数式 setState、组件顶层定义、事件处理器中执行副作用等涉及逻辑正确性（而非仅是性能）的规则仍需严格遵守。

## 运维、部署与可用性规范
1. 系统必须具备一键维护状态切换能力。通过中间件读取全局配置，一旦开启维护模式，必须在网络层拦截所有非白名单请求，并将其重定向至专属的维护说明页面，实现零代码修改的热切换。
2. 所有的部署环境必须通过独立的环境变量文件进行隔离管理。核心机密信息（如密钥、盐值、第三方服务令牌）必须由宿主机的环境变量注入，严禁随代码提交至版本控制系统。
3. 系统必须具备全局的未捕获异常处理机制。无论是前端的渲染崩溃还是服务端的运行异常，都必须被边界组件或全局处理器拦截，并向用户展示友好的降级界面，同时记录详细的错误日志。

## 日志与监控预警规范
1. 必须建立结构化的全局日志系统，严禁在生产环境中使用原生的控制台输出函数。所有日志必须包含时间戳、日志级别、请求上下文追踪标识以及操作者身份标识。
2. 日志级别必须严格分级：致命错误、普通错误、警告、信息与调试。生产环境默认仅输出信息及以上级别的日志，避免海量调试信息打满磁盘。
3. 严禁在日志中打印敏感信息（如用户密码、会话凭证、完整支付信息等），所有流入日志系统的数据必须经过脱敏清洗拦截器。
4. 所有的服务端核心异常、未捕获的全局错误以及关键的安全拦截（如越权访问尝试、高频恶意请求）必须触发错误级别日志，并自动接入实时告警系统。
5. 日志写入操作必须采用异步非阻塞模式，确保高并发场景下磁盘写入阻塞不会拖垮主业务进程的响应速度。

## 文档、注释与代码质量规范
1. 项目文档必须实行国际化双语管理，在文档根目录下严格划分为中文版与英文版两个独立目录，所有核心架构指南与部署说明必须保持双语同步更新。
2. 严禁文档与代码脱节。必须在持续集成流程或代码提交的钩子中接入自动化提取脚本，在代码提交前自动扫描目录树和架构分析并覆盖更新说明文档，确保项目文档时刻保持最新。
3. 当问题解决或代码修改完成后，开发者必须额外检查一遍所有被修改过的文件中的注释，确认注释是否已经补齐。
4. 注释必须使用简体中文。注释内容必须与当前代码保持绝对一致，严禁存在过时、错误、缺失或容易误导他人的废弃注释。
5. 在任何环节中，包括但不限于代码执行、终端输出、编译、打包、命令运行等过程，均不允许出现任何错误提示或警告信息。一旦出现，必须立即定位并修正其根源。
6. 对于任何警告或错误，绝对不允许采用关闭日志、降低警告等级、绕过检查或其他掩盖问题的方式处理。必须修复问题本身，而不是隐藏问题表现。

## AI协同开发（Vibe Coding）规范
1. 为保证人工智能模型在代码生成与协作过程中能够严格遵循既定的项目机制，必须在项目根目录设立模型专属的系统指令约束文件。
2. 必须将本开发规范中的核心原则（如禁用硬编码、强制异常捕获、严格的文件命名法、双轨制接口设计等）以指令形式硬性注入约束文件中。
3. 确保模型在每次生成代码前，都会自动加载并应用这些上下文约束，使得自动生成的代码在架构美学、健壮性和安全性上完全贴合本项目的顶级要求，实现无缝的人机结对编程。
